## 0. 一页纸速览（TL;DR）

**续（Continuator）是一条本地运行的"小说续写生产线"**：喂进一本已出版的小说（`.txt` / `.epub`），它自动抽取人物·关系·世界观设定，规划后续情节，逐章生成正文，并用一个多视角 AI 评审团把关质量与原著保真度——整个过程可以无人值守地连续推进几十章，每一步都有成本账本和质量门禁。

| 维度 | 一句话 |
|---|---|
| **它是什么** | 把"编辑 / 策划 / 写手 / 审校"四种角色拆成多个 LLM agent 协作的续写流水线 |
| **解决什么** | 直接让 ChatGPT 写一章常见的：风格飘移、人物 OOC、伏笔遗忘、剧透穿越、逻辑断裂 |
| **怎么用** | 一条 CLI 命令链，或浏览器里的"小白四步工作台" |
| **跑在哪** | 全本地：源作品原文不出仓库，生成稿也只落本地 |
| **现在到哪了** | 九阶段流水线全部打通；龙族、原创 premise 书等多部作品真实验证过；正在攻关"深卷起点"的防剧透机制 |
| **技术底色** | Python 3.9+ / LiteLLM 多模型路由 / Pydantic / 标准库 Web，**无 async、无 Web 框架、无编排库** |

## 1. 产品定位

### 1.1 它解决的真实痛点

把一本小说丢给通用聊天模型说"继续写第 N 章"，你大概率会撞上这几堵墙：

- **风格飘移**——越写越不像原作者的笔触；
- **人物 OOC**——角色说出不符合其性格的话、做出不符合其处境的事；
- **伏笔与设定遗忘**——几百页前埋的线索、人物关系，模型记不住；
- **剧透穿越**——模型从预训练记忆里掏出"原著后面才发生的事"，提前写进续作，破坏续写起点的合理性；
- **逻辑断裂**——多章之间时间线、因果关系对不上。

根因是单次对话**没有持久的世界观记忆，也没有独立的质检环节**。续把这两件事工程化了。

### 1.2 目标用户

- **创意写作者 / 同人作者**：想在一本喜欢的小说某个节点接着写下去；
- **原创作者**：用一句话立意（premise）起一本新书，让系统帮忙扩写设定、规划情节、产出初稿；
- **多 agent 协作研究者**：这套系统本身是一个"多 LLM agent 分工 + 结构化决策 + 质量门禁"的完整工程案例。

> 使用前提：你对源作品有合法访问权。本产品定位为**本地私有工具**，原文与生成内容均不上传。

## 2. 核心差异化：为什么不直接用 ChatGPT

| | 直接让通用模型续写 | 续（Continuator） |
|---|---|---|
| **世界观记忆** | 靠对话上下文，长了就忘 | 自动抽取并持久化人物图谱 / 设定库 / 关系状态，每章写作都基于完整知识库 |
| **质量把关** | 模型自评，倾向于"放行" | 5+1 个独立视角的 AI 评审团 + 确定性 linter，**fail-closed**（评审解析失败按弃权处理，绝不默认放行） |
| **防剧透** | 几乎无法控制 | 三道起点一致性护栏 + 物理截断摄入，从数据源头杜绝"原著后续内容泄露进续作" |
| **连续产出** | 一章一章手工喂 | 长程驱动器无人值守连写几十章，失败自动重写、关系自动推进、断点续跑零重复花费 |
| **成本可控** | 不透明 | 每次调用记 token 与单价，按 provider 汇总，预算超限自动停机 |
| **隐私** | 内容进第三方服务 | 全本地，原文不出仓库 |

一句话：**续把"让 AI 写得像、记得住、不剧透、还能批量产出且不烧钱"这件事，做成了一条有门禁的流水线，而不是一次 prompt。**

## 3. 产品能力地图：9 阶段续写流水线

一条续写指令从输入到成稿，经过 9 个阶段。每个阶段都是独立可重跑的，产物落盘、可人工编辑后再进入下一步。

| # | 阶段 | 输入 → 输出 | 它做了什么 |
|---|---|---|---|
| **1** | **输入准备** | 原始 `.txt`/`.epub` → 标准化文本 + 章节清单 | 编码自动检测与规范化；中英文自动判定；按对应规则切章（中文「第N章」/ 英文「CHAPTER N」/ POV 块），带章节置信评分。EPUB 用标准库直转，零新依赖 |
| **2** | **知识抽取** | 采样章节 → 全局知识库 + 实体图谱 | 分句提取实体 / 关系 / 事实 / 证据片段；压缩为全局知识库；一键产出 5 类提案（事实库 / 实体图谱 / 起点锚点 / 风格样例 / agent 角色卡），**人工确认后才落盘** |
| **3** | **起点判断** | 用户选定续写起点章 → 起点指纹 | 把"从哪一章之后开始续写"设为硬门；生成起点指纹（章号 + 内容哈希）用于后续一致性校验 |
| **4** | **关系/世界观激活** | 知识库 → 按起点过滤的"安全视图" | 加载实体图谱、事实库、角色卡，并**过滤掉起点之后的剧透内容**（含按人物 POV 视角过滤），喂给下游的只有"起点时刻该知道的事" |
| **5** | **情节规划** | 起点 + 知识库 → 分章细纲（图纸） | 6 个 agent 多轮辩论续写方向 + 结构化投票；强模型据此一次性规划 N 章细纲，含开篇 / 结尾预览 |
| **6** | **写作生成** | 细纲 + 知识库 + 原文衔接片段 → 章节正文 | 生成 3500–5500 字正文，内置自检改写循环（最多 3 轮）；读取知识库 / 事实 / 实体状态 / 细纲 / 滚动摘要 / 风格样例 |
| **7** | **审核评审** | 正文 → 评审结论 + 修改意见 | 5+1 agent 评审面板按"情节 / 文笔 / 原著保真"三维打分；确定性 linter 查句式硬伤；关系审计零成本查一致性；**任一评审给出实质性 Reject → 整章 Reject** |
| **8** | **关系更新** | 通过的章节 → 更新后的实体图谱 | 自动提议人物关系/状态的推进，高置信度的自动应用，让世界观随剧情演进 |
| **9** | **滚动到下一章** | 已发布章节 → 滚动摘要 + 成本报告 | 维护跨章滚动摘要（分层：上层摘要 + 最近 K 章原文片段），作为下一章的上下文；汇总本轮 token 与成本 |

> 这是一份**活文档化的 SOP**：每个节点的打通状态在 README 的「流水线 SOP（实时状态）」里逐轮同步。

## 4. 质量保障体系（产品的硬核差异化）

这是续区别于"调个 API 循环写"的核心。质量不是靠某个更强的模型，而是靠一套**多层防御 + 门禁**。

### 4.1 三道起点一致性护栏

防止"续写内容与你选定的起点对不上"或"用了过期的规划/大纲"：

1. **细纲 ↔ 起点**：章节细纲必须携带与起点一致的指纹（起点章号 + 内容哈希），不一致直接硬拦。
2. **大纲 ↔ 起点**：辩论产出的大纲带元数据（起点章号 / 起点指纹 / 大纲哈希），规划前校验，指纹不匹配硬拦；提供 `--allow-stale-outline` 显式逃生门并留审计。
3. **知识库按起点截断**：所有消费 prompt（写手 / 辩论 / 评审 / 规划）一律通过"起点安全视图"过滤，只注入章号 ≤ 起点的知识——知识库压缩时故意做成超集，消费层再收窄。

### 4.2 ingest-to-start 物理截断摄入（"结构性免疫"）

针对**新上传的书**：上传起点后，摄入 → 规范化 → 切章时**直接丢弃起点之后的所有章节**，本地规范化文本物理上只剩"起点及以前"。

这样一来，几条最危险的"逐字泄露口"（风格样例 / 原文摘录 / 实体关键事实）和重复摄入问题**从数据源头消失**，下游不需要任何"要不要过滤"的条件判断——因为物理上根本不存在起点后的内容。这就是"结构性免疫"：不靠 `if` 兜底，靠数据源头根除风险。

> 既有的全本 workspace（如 longzu）继续走 §4.1 的过滤护栏作为兜底；两条路径互补。

### 4.3 diff oracle（过滤正确性的机器证明）

怎么证明"防剧透真的有效"？续用一个**机器对照**：同一个起点跑两遍——(A) 物理截断模式 vs (B) 全本 + 过滤模式——然后逐字节 diff 两边注入给写手/辩论的全部材料（知识库 / 实体状态 / 原文摘录 / 风格样例 / 锚点）。

若两边逐字节一致，即从机器层面证明过滤无泄露。当前已对最严重的泄露口（风格样例 / 逐字原文）验证两模式逐字节一致，起点后标志性事件在两边均为零出现。**这把"人眼看有没有穿越"换成了"机器证明没有穿越"。**

### 4.4 三视角 AI 评审面板（5+1 agent，fail-closed）

每章生成后由一个评审团把关：

- **5 个投票者**覆盖三个维度——情节一致性、文笔质量、原著保真度（外加一个只给建议、不投票的顾问角色）；
- **1 个确定性关系审计**：纯程序规则查人物关系一致性，**零 LLM 成本**；
- **三维独立打分**，可分化（曾实测到"保真度"与"文笔"分差 4–8 分，说明三个轴真的在分别度量）；
- **fail-closed**：评审返回的 JSON 解析失败 → 记为弃权（Abstain）而非默认通过；
- **结构化拒因回灌**：block 级（禁令）问题与可选优化建议分层回灌给写手，让它能区分"必须改"和"可以改"。

配合按门槛分档（见 §6 tier），评审面板是"宁可拦错、不可放过"的最后一道闸。

### 4.5 确定性 linter + 成本护栏

- **linter**：用确定性规则查句式硬伤（如滥用句式、字数约束），零成本、零随机性；
- **预算护栏**：写作与长程驱动均支持 `--budget-cny N`，事后结算超限即停机（退出码 3），不继续扣费；真模型运行前必过 **preflight** 铁闸（环境 / provider 路由 / 上下文上限 / 清单完整性等 FATAL 检查）。

## 5. 系统架构与技术栈

### 5.1 子系统一览（`src/`，约 46 个模块）

| 分组 | 关键模块 | 职责 |
|---|---|---|
| **核心流水线** | `extractor` / `compressor` / `debater` / `plot_planner` / `writer` / `reviewer` | 抽取 → 压缩知识库 → 辩论 → 规划 → 写作 → 评审 |
| **质量门禁** | `linter`（+`lint.yaml`）/ `relationship_auditor` | 确定性句式检查 / 零成本关系一致性审计 |
| **知识管理** | `entities` / `entity_advance` / `kb_view` / `persona_loader` / `style` | 实体图谱 / 关系推进 / 起点安全视图 / 角色卡 / 风格样例 |
| **多语言·多书** | `lang_detect` / `chapter_splitter` / `epub_to_txt` / `paths` | 中英判定 / 切章 / EPUB 转换 / 多 workspace 路径 |
| **自动化编排** | `auto_bootstrap` / `auto_pipeline` / `book_driver` / `book_runner` | 一键产提案 / 9 步 SOP 编排 / 长程驱动器 / 生产级写书 runner |
| **守门·工程** | `preflight` / `config` / `llm_client` / `cost_estimator` / `schemas` | 起飞前检查 / 配置 / 真假模型抽象 / 成本汇总 / Pydantic schema 唯一真源 |
| **辅助规划** | `premise_expansion` / `drama_planner` / `hook_designer` / `foreshadowing` / `continuation_anchor` / `chapter_summary` | 立意扩写 / 戏剧·钩子·伏笔规划 / 起点锚点 / 滚动摘要 |

### 5.2 Web 工作台

- **零前端依赖**：纯标准库 `http.server` + `string.Template` + 原生 JS，没有任何前端构建链；
- **小白四步工作台**：设定 → 大纲 → 细纲 → 正文，四阶段卡片式向导，一句话即可开新书（premise → 异步 job 轮询）；
- **全程可编辑**：每个阶段的产物都能内联编辑保存，并有 mtime 链自动失效管理（改了上游，下游自动标记过期）；
- 书架首页 + 多页信息架构（设置 / 计划 / 细纲 / 章节 / 评审 / 成本），移动端侧栏收折、宽表横向滚动。

### 5.3 技术栈

| 层 | 选型 |
|---|---|
| 语言 | Python 3.9+ |
| 多模型路由 | [LiteLLM](https://github.com/BerriAI/litellm)（OpenAI 兼容 / DeepSeek / Claude 等） |
| Schema | [Pydantic](https://docs.pydantic.dev/)（配置 + 数据校验的唯一真源） |
| Token 计数 | [tiktoken](https://github.com/openai/tiktoken)（回落 `cl100k_base`） |
| 环境 | [python-dotenv](https://github.com/theskumar/python-dotenv) |
| Web | 标准库 `http.server`，纯 Python + JSON I/O |

**刻意不用**：无 async 框架（全同步）、无 Web 框架、无编排库——编排就是 Python 循环。设计哲学是"少依赖、可审计、好排错"。

## 6. 运行模式

### 6.1 三种模式

| 模式 | 用途 | 特点 |
|---|---|---|
| **mock（默认）** | 开发 / 单测 / 快速验证流程 | 零 token、不联网、完全确定性；写作返回固定短稿（故意触发 linter 拦截，以检验门禁本身）；近千例单测秒级跑完 |
| **真模型** | 生产续写 | 先过 preflight 铁闸；按 tier 选质量档；每次调用记成本；支持预算护栏与 per-task 模型覆写（写手/辩论/评审/规划可各配各的模型） |
| **长程驱动器** | 无人值守批量续写 | `drive-book` 子进程脱离会话后台跑（防睡眠）；进度永远从盘面推导（防双真源）；遇质量闸 blocked 停下等人审；stop/resume 断点续跑**零重复花费** |

### 6.2 质量档（tier）

写作/评审按门槛分三档，对应不同严格度：

| Tier | 投票门槛 | 总分线 | 文笔下限 | 适用 |
|---|---|---|---|---|
| **high** | 5/5 | 8.0 | ≥6 | 出版级 |
| **mid** | 4/5 | 7.5 | ≥6 | 生产默认 |
| **low** | 3/5 | 6.5 | ≥5 | 快速试验 / 草稿 |

### 6.3 模型后端

通过 LiteLLM 路由，支持任意 OpenAI 兼容后端；已实测包括 GPT‑5.5 系列、DeepSeek、Claude 系列。生产用 `.env` 配 `OPENAI_*`，可为 planner/writer/reviewer/debater 分别覆写更强或更省的模型。

## 7. 使用方式

### 7.1 命令行（以续写《龙族》为例）

```bash
# 1) 建 workspace，放入原文（或从 EPUB 导入）
python3 main.py workspace-init longzu
python3 main.py --book longzu epub-import --src ~/longzu.epub   # 或直接放 txt

# 2) 标准化 + 切章（语言自动判定）
python3 main.py --book longzu normalize
python3 main.py --book longzu split

# 3) 一键抽设定，产 5 类提案，人工看过后 confirm
python3 main.py --book longzu init-book --extract-limit 10
#   apply-bootstrap --name {global_facts|entity_graph|continuation_anchor|style_examples|personas} --confirm

# 4) 定起点 → 辩论方向 → 规划细纲
python3 main.py --book longzu set-start-point <chapter_id>
python3 main.py --book longzu debate
python3 main.py --book longzu plan-chapters --chapters 3 --force

# 5) 就绪检查 → 开写
python3 main.py --book longzu write-readiness --chapters 3
bash scripts/write_book.sh --book longzu --chapters 3 --tier mid --budget-cny 10

# 进度与成本
python3 main.py --book longzu chapter-status <n>
python3 main.py estimate-cost
```

### 7.2 浏览器

```bash
python3 main.py web      # 默认 http://127.0.0.1:8765
```

打开后从书架进入"小白四步工作台"，按设定 → 大纲 → 细纲 → 正文逐步推进，每步可编辑可重跑。

### 7.3 一本书一个 workspace

`workspaces/<name>/` 完全隔离，靠 `--book` 切换，彼此零干扰。当前仓库内已并存 longzu（龙族）、shudian052（原创 premise 书）、tianlong、i38drama01 等多部。

## 10. 术语速查

| 术语 | 含义 |
|---|---|
| **起点指纹（fingerprint）** | 起点章号 + 内容 SHA256，用于规划/大纲/知识库的一致性校验 |
| **起点安全视图（start_safe_knowledge）** | 只注入"章号 ≤ 起点"的知识库视图，下游 prompt 的统一入口 |
| **ingest-to-start 截断** | 摄入时物理丢弃起点后章节，从源头免疫剧透泄露 |
| **diff oracle** | 截断模式 vs 过滤模式逐字节对照，机器证明防剧透有效 |
| **bootstrap / proposal** | 抽取后产出的 5 类待确认提案（事实 / 图谱 / 锚点 / 风格 / 角色卡） |
| **debate outline** | 6 agent 辩论产出的情节方向大纲，规划细纲的种子 |
| **chapter_plan（细纲/图纸）** | 强模型据大纲与起点生成的分章细纲，写手的施工图 |
| **rolling_summary（滚动摘要）** | 分层的跨章摘要（上层摘要 + 最近 K 章原文片段），每章更新 |
| **entity advance** | 人物关系/状态的推进提案，评审后可自动应用 |
| **tier（low/mid/high）** | 评审门槛档位，决定投票数 / 分数线 / 文笔下限 |
| **fail-closed** | 评审异常时按"不放行"处理（弃权而非默认通过） |
| **preflight** | 真模型运行前的 FATAL/WARN 分级起飞检查 |
| **drive-book（长程驱动器）** | 脱离会话后台连写多章，断点续跑零重复花费 |
